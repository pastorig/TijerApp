## Architecture Overview

Una **sola PWA "TijerApp"** que cubre toda la plataforma multi-tenant. El barbero o cliente que instala desde cualquier ruta (`/[slug]/admin` o `/[slug]`) obtiene el mismo icon en su home screen, y al abrir vuelve al **último contexto navegado** vía `localStorage` + página redirector en el `start_url`.

Implementación apoyada en **`@serwist/next`** para el service worker (modern successor de next-pwa, App Router-first). El SW aplica strategy-per-route caching: HTML como `StaleWhileRevalidate`, assets versionados como `CacheFirst`, API/Supabase como `NetworkOnly`. Cuando una request HTML falla y no hay cache → fallback a `/offline` (ruta estática pre-cacheada).

El install prompt es **híbrido**: un Context Provider en `app/layout.tsx` captura el `beforeinstallprompt` event globalmente y expone un hook `usePWAInstall()`. Componentes `InstallButton` se montan en footer admin y banner en landing pública — invisibles cuando la app ya está instalada (`display-mode: standalone`). En iOS Safari, donde no hay `beforeinstallprompt`, el botón muestra un tooltip con instrucciones manuales.

## Stack Decisions

| Concern | Decision | Rationale |
|---|---|---|
| Service Worker | `@serwist/next` | Successor oficial de next-pwa, App Router-native, mantenido |
| Icon generation | `npx pwa-asset-generator` one-time | One-shot tool, output commiteado, sin build dep |
| Last-context storage | `localStorage` con prefix `tijerapp:` | Simple, client-only, sobrevive instalación |
| Cache strategy | strategy-per-route via serwist runtime caching | Balance freshness vs speed |
| Offline fallback | Ruta estática `/offline` | Pre-cacheable, sin deps dinámicas |
| Install prompt UX | Hybrid: nativo via Context + custom buttons | Maximiza discoverability; iOS fallback con tooltip |
| Manifest declaration | `app/manifest.ts` (Next.js declarative) | Type-safe, integrado con metadata Next 16 |

### Key Interactions

| User action | Sistema |
|---|---|
| Navega `/sv-barber/admin` | `useLastContextTracker` guarda `tijerapp:last_slug=sv-barber, last_role=admin` |
| Browser dispara `beforeinstallprompt` | Provider captura, `isInstallable=true`, InstallButton aparece |
| Click "Instalar app" | `prompt()` nativo del browser → user accepts/dismisses → `appinstalled` event |
| Tap icon del home screen | Browser abre `start_url: /?source=pwa` → root page lee localStorage → redirect a `/<slug>/admin` |
| Sin conexión + tap icon | SW falla request HTML → serve `/offline` desde cache |
| User en iOS Safari + click "Instalar" | Tooltip explica "Tocá Compartir ↑ → Agregar a inicio" |

## Testing Strategy

- **Lint + build verification**: `npm run lint && npm run build` → ambos clean.
- **Lighthouse PWA score**: ≥ 90 en producción (run after deploy). Tareta: ≥ 95.
- **Manual smoke tests** (en este orden):
  1. **Dev local**: `npm run dev` → ver manifest en DevTools Application tab.
  2. **Build local**: `npm run build && npm start` → SW se registra, ver en DevTools → Application → Service Workers.
  3. **Install desktop**: Chrome desktop → debería aparecer el icon ⊕ en la URL bar.
  4. **Install Android**: Chrome mobile → menú 3-puntos → "Instalar app".
  5. **Install iOS**: Safari → compartir → "Agregar a inicio" → verificar icon.
  6. **Offline test**: DevTools → Network → Offline → recargar → debe mostrar `/offline`.
  7. **Last-context test**: navegar `/sv-barber/admin` → cerrar tab → abrir desde home screen → debe ir a `/sv-barber/admin`.
- **Edge cases a verificar**:
  - Navegación entre slugs (`/sv-barber` → `/ag-barber`): el `last_slug` se actualiza al último.
  - Dismiss banner en landing: no reaparece por 30 días.
  - Browser sin SW support (Safari < 16.4): app sigue funcionando como web normal.

## Risks & Mitigations

| Risk | Severity | Mitigation |
|---|---|---|
| `@serwist/next` rompe Next.js dev server (`npm run dev`) | High | Configurar serwist `disable: process.env.NODE_ENV === "development"` para dev. SW solo en prod. |
| Service Worker cachea HTML stale después de un deploy | Medium | Strategy `StaleWhileRevalidate` para HTML — siempre re-fetch en background, nuevo deploy se ve al refresh siguiente. |
| iOS Safari ignora `start_url` query param | Low | Tolerable; el fallback es `/` que es válido como home. |
| User instala desde landing pública pero no quiere ver admin | Low | El last_role distingue `admin` vs `public`; solo redirect a admin si ese es el último contexto. |
| pwa-asset-generator falla con SVG con strokes | Medium | Probar antes de implementar; si falla, generar con sharp o ImageMagick local. |
| Service Worker registration error en prod (Vercel headers, etc.) | Medium | Verificar serving de `/sw.js` con `Service-Worker-Allowed: /` header. Documentado en quickstart. |
| User clears localStorage → last-context se pierde | Low | Aceptado; fallback al home de la plataforma es válido. |
| El isotipo no es visualmente óptimo a 16px favicon | Low | Ya fue verificado visualmente en sesión previa. |

## Rollback Plan

Si la feature rompe producción tras el deploy:

1. **Revert commit del merge** en `main`:
   ```bash
   git revert -m 1 <merge-commit-sha>
   git push origin main
   ```
2. **Unregister service worker** en clientes que ya tienen la versión rota:
   - El revert deja un SW vacío. Los clientes existentes auto-unregister en su siguiente visita.
   - Para forzar: agregar header `Clear-Site-Data: "cache","cookies","storage"` temporal en `next.config.ts` para `/`.
3. **NO migration rollback necesario** (sin DB changes).
4. **Disable PWA via env flag** (preventivo): no implementado en este plan, pero si en el futuro queremos un kill switch, env var `NEXT_PUBLIC_PWA_DISABLED=1` que el SW respeta.
