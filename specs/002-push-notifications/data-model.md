# Data Model: Push Notifications

**Feature**: [spec.md](spec.md)
**Plan**: [plan.md](plan.md)
**Date**: 2026-06-04

## Entities

### `push_subscriptions`

Representa un device (browser instance) suscripto a notificaciones para una barbería.

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | uuid PK | ✓ | Generated by DB |
| `created_at` | timestamptz | ✓ | When subscription was first created |
| `last_used_at` | timestamptz | ✓ | Updated each successful send; used for cleanup heuristics |
| `expired_at` | timestamptz | nullable | Set when we detect 410 Gone from push service. Active rows have `expired_at IS NULL`. |
| `barbershop_slug` | text | ✓ | FK → `barbershops(slug)` ON DELETE CASCADE |
| `user_id` | uuid | ✓ | FK → `auth.users(id)` ON DELETE CASCADE |
| `endpoint` | text | ✓ | URL del push service del browser (FCM/Mozilla/Apple) |
| `p256dh` | text | ✓ | Public key del browser para encryption |
| `auth` | text | ✓ | Auth secret del browser |
| `user_agent` | text | nullable | UA string captured at subscribe time (debugging) |

**Constraints**:
- `UNIQUE (user_id, endpoint)` — mismo device no se duplica si re-subscribe

**Index**:
- `(barbershop_slug) WHERE expired_at IS NULL` — para el trigger que lee subs activas

**State transitions**:
```
created (expired_at=null)
   │
   ├── manually unsubscribed by admin → DELETE
   │
   ├── 410 Gone detected during send → UPDATE expired_at = now()
   │
   └── successful send → UPDATE last_used_at = now()
```

**Lifecycle rules**:
- Subscription se crea cuando admin acepta browser permission y guardamos en server
- Se borra (`DELETE`) cuando admin desactiva desde UI
- Se marca `expired` (no DELETE) cuando el endpoint devuelve 410 — preserva trail
- Cleanup automático borra rows con `expired_at < now() - 30 days`

### `push_notification_queue`

Item de notificación pendiente de enviar.

| Field | Type | Required | Notes |
|---|---|---|---|
| `id` | uuid PK | ✓ | Generated by DB |
| `created_at` | timestamptz | ✓ | Cuando se encoló (idealmente <1s del appointment insert) |
| `sent_at` | timestamptz | nullable | Cuando se procesó exitosamente |
| `subscription_id` | uuid | ✓ | FK → `push_subscriptions(id)` ON DELETE CASCADE |
| `payload` | jsonb | ✓ | `{ title, body, url, tag }` que va al browser |
| `status` | text | ✓ | `pending` / `sent` / `failed` / `invalid` |
| `retry_count` | int | ✓ | Default 0; incrementa en cada retry |
| `last_error` | text | nullable | Mensaje del último error (debugging) |

**Status state machine**:

```
                ┌─→ sent  (delivery 200/201)
                │
   pending  ────┼─→ invalid  (410 Gone — subscription dead)
                │
                └─→ failed  (5xx after 3 retries OR 4xx non-410)
```

**Index**:
- `(status, created_at) WHERE status = 'pending'` — para el cleanup que busca stuck items

**Payload shape**:
```typescript
type Payload = {
  title: string;        // "Nueva reserva"
  body: string;         // "Juan · 18:30 con Carlos"
  url: string;          // "/sv-barber/admin/turnero" — ruta a abrir en tap
  tag: string;          // "appointment-<uuid>" — para dedup en notification tray
}
```

**Retention**:
- Items `sent` o `failed` se borran después de 7 días (housekeeping)
- Items `invalid` se borran junto con la `push_subscriptions` row que les dio origen

## Relationships

```
auth.users           barbershops
     │                    │
     │                    │
     │ user_id            │ barbershop_slug
     ▼                    ▼
push_subscriptions  ◄──── (1:N) ──── (1:N) ────────────► appointments
     │                                                        │
     │ id                                                     │ AFTER INSERT trigger
     │                                                        │
     │ subscription_id                                        │
     ▼                                                        ▼
push_notification_queue ◄─── inserts via enqueue_push_for_appointment()
```

## Validation Rules

- `endpoint` debe ser HTTPS URL válida (no validamos en DB; el browser nunca debería dar otra cosa)
- `payload.title` y `payload.body` requeridos, max 50 + 200 chars (Apple/Chrome rendering limits)
- `payload.url` debe ser un path relativo empezando con `/` (no full URL)
- `retry_count` ≤ 3 (más allá de eso, mover a `failed`)

## Triggers

### `enqueue_push_for_appointment()` (AFTER INSERT on appointments)

**Behavior**:
1. Si `NEW.status NOT IN ('pending','confirmed')` → no-op (return NEW)
2. Construye payload con `title="Nueva reserva"`, body con cliente+hora+barbero
3. Para cada `push_subscription` activa de la barbería (`expired_at IS NULL`):
   - Insert row en `push_notification_queue` con status='pending'

**Atomicity**: trigger AFTER INSERT corre en la misma transacción que el INSERT del appointment. Si la transacción rollback (constraint violation, etc.), las queue rows también se rollback. Garantiza "no notif sin appointment".

## RLS Policies

| Table | Role | Operation | Policy |
|---|---|---|---|
| `push_subscriptions` | authenticated | SELECT | `user_id = auth.uid()` |
| `push_subscriptions` | authenticated | INSERT | `user_id = auth.uid() AND current_user_has_barbershop_access(barbershop_slug)` |
| `push_subscriptions` | authenticated | DELETE | `user_id = auth.uid()` |
| `push_subscriptions` | authenticated | UPDATE | bloqueado (las updates las hace service_role desde el cleanup) |
| `push_notification_queue` | any auth | ALL | bloqueado por default (RLS enabled sin policies) |
| `push_notification_queue` | service_role | ALL | bypass RLS (inserts via trigger SECURITY DEFINER, reads desde processor) |

## Cleanup Policies

Ejecutadas por `/api/push/cleanup` (GitHub Actions hourly):

| Condition | Action |
|---|---|
| `queue.status='pending' AND created_at > 30 min ago` | Reintentar send (incrementa retry_count) |
| `queue.status='pending' AND retry_count >= 3` | Marcar como `failed` |
| `queue.status IN ('sent','failed') AND created_at > 7 days ago` | DELETE row |
| `subscriptions.expired_at < 30 days ago` | DELETE row |
